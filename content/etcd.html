etcd

The Kubernetes deep-dive ended on one sentence worth re-reading: "nothing but the API server ever talks to etcd directly." That sentence hides the single component whose failure takes down a Kubernetes cluster's ability to accept any write at all. Every Pod spec, every Secret, every ConfigMap, every controller's view of "current state" — all of it lives in etcd, and every one of those reads and writes has to survive network partitions, crashed nodes, and disk hiccups without ever returning two different answers to the question "what does the cluster look like right now?" This deep-dive is about how etcd delivers that guarantee, and what it costs to run.

Where this fits etcd graduated CNCF in November 2020. It's a general-purpose, strongly consistent distributed key-value store — nothing about it is Kubernetes-specific — but it became CNCF's canonical consensus-backed store because Kubernetes chose it as the one place all cluster state lives. Its own etcd-io/raft Go library, factored out as a standalone module, is independently reused by CockroachDB, TiDB, Project Calico, and Flannel — arguably the most widely deployed production Raft implementation in existence, on top of etcd itself.

The problem: agreeing on one truth across unreliable machines

A single database on a single machine never has to ask "which of us is right?" A distributed store that replicates data across several machines for durability and availability constantly does — a network partition or a crashed leader can leave two nodes each convinced they're in charge, and if both accept writes, the cluster now has two histories that disagree. The algorithm that prevents this is called consensus: get a set of nodes to agree on a single, ordered sequence of operations, even when some nodes crash or messages are delayed or lost, as long as a majority of nodes can still talk to each other.

The first practical consensus algorithm, Paxos, is notoriously hard to reason about and even harder to implement correctly. etcd is built on Raft, a consensus algorithm published in 2014 with an explicit design goal: be understandable without sacrificing the same safety guarantees. Raft does this by decomposing the problem into three separable pieces — leader election, log replication, and safety — each with a small, checkable set of rules. What follows is exactly those three pieces, as etcd implements them.

Raft: electing a leader and replicating a log

Every etcd node (a "member") is at all times in exactly one of three roles: follower, candidate, or leader. All writes go through the leader; followers only ever replicate what the leader tells them. Time is divided into terms, each identified by an increasing integer, and at most one leader can be elected per term.

  • Election. A leader sends empty AppendEntries RPCs as heartbeats to every follower on a fixed interval. Each follower resets a randomized election timeout every time it hears from the leader. If a follower's timeout fires — because the leader crashed, was partitioned away, or was just slow — it increments its term, votes for itself, becomes a candidate, and requests votes from every other node. A candidate that collects votes from a majority of the cluster for that term becomes the new leader. The randomized part of the timeout matters: if every follower had the same timeout, they'd all become candidates simultaneously and split the vote every single time.
  • Log replication. Once a leader exists, every client write is appended to the leader's local log first, then replicated to followers via AppendEntries RPCs carrying real entries instead of empty heartbeats. The leader tracks, per follower, a nextIndex (what to send next) and matchIndex (the highest entry confirmed stored on that follower). Once an entry is durably stored on a majority of members — including the leader — the leader advances its commit index and applies the entry to its state machine (etcd's key-value store). Followers learn the new commit index on the next heartbeat and apply it locally. A write is never acknowledged to the client until it's on a majority of disks — that's what makes it durable against losing any minority of the cluster.
Quorum: why odd-sized clusters A cluster of n members needs a quorum of floor(n/2)+1 nodes to elect a leader or commit a write. Because quorum size doesn't shrink going from an even n down to n-1, an even-sized cluster buys zero extra fault tolerance for an extra machine — a 4-node cluster needs the same 3-node quorum as a 3-node cluster, but tolerates the same single failure. That's why every etcd cluster you'll see in the wild is sized 3, 5, or 7.

The data model: MVCC on top of a B+tree

Below Raft, etcd stores its actual key-value data using MVCC on top of bbolt (the maintained fork of BoltDB), a single-file, memory-mapped B+tree that gives etcd local ACID transactions. Every write to the store — a put or delete, for any key — increments one global, monotonically increasing revision number for the entire keyspace. The actual bbolt row key isn't the user's key at all; it's (revision, sub-revision), with the sub-revision disambiguating multiple keys written in the same transaction. A separate in-memory index (etcd calls it the keyIndex) maps each logical key name to all of its historical revisions, so a request for "key X as of revision R" is a lookup, not a scan.

This is exactly what makes Kubernetes' resourceVersion mean something precise: it is etcd's revision number, passed straight through the API server. "Get this Deployment at resourceVersion 4021" is "get this key at etcd revision 4021" — a genuinely consistent, point-in-time read, not a best-effort snapshot.

Compaction and defrag are two different things Because MVCC keeps every historical version of every key, etcd runs periodic compaction (Kubernetes' API server triggers one roughly every 5 minutes) to discard old revisions before a cutoff. But bbolt is copy-on-write — compaction frees internal pages onto a freelist for reuse, it does not shrink the file on disk. Only etcdctl defrag rebuilds the file to actually return space to the filesystem, and running it live blocks reads/writes on that member while it works — which is why operators defrag one member at a time, in a rolling fashion, never the whole cluster at once. Skip both long enough on a busy cluster and you'll eventually see mvcc: database space exceeded.

Watch: how controllers subscribe to change without polling

Recall from the Kubernetes deep-dive that every controller registers interest in a resource type and gets pushed deltas instead of polling. That mechanism is etcd's watch, and it exists because of the MVCC revision model above. A client opens a long-lived gRPC Watch stream, specifies a key or range and a starting revision, and etcd streams back ordered PUT/DELETE events from that revision forward as Raft commits them — genuinely pushed, not polled. Because every historical revision is (until compacted) still indexed, a client that reconnects can resume exactly where it left off by watching from its last known revision; if that revision has already been compacted away, etcd returns an error and the client has to re-list to resync — the same shape as Kubernetes' "410 Gone: resourceVersion too old" response to a stale watch.

Kubernetes doesn't have each controller open its own etcd watch, though — the API server runs one internal watch cache per resource type (a single long-lived List+Watch against etcd, fanned out in-memory to every client's own watch), which is exactly the layer that turns one etcd stream into thousands of concurrent kubectl get -w sessions without multiplying load on etcd.

Leases: a TTL primitive for heartbeats and leader election

A lease is a first-class etcd object: a client calls Grant(ttl) to get a lease ID, attaches it to any number of keys, and keeps it alive with a bidirectional LeaseKeepAlive gRPC stream. Stop renewing, and every key on that lease is deleted atomically when the TTL runs out — turning "is this thing still alive" into a property etcd itself enforces, rather than something a client has to poll for.

Kubernetes' coordination.k8s.io/v1 Lease objects are a thin layer over exactly this: kubelets renew a per-node lease on a fast interval so node-death detection doesn't require diffing full Node objects, and HA control-plane components (the scheduler, controller-manager) use a lease the same way a distributed lock works — whichever replica holds and renews it is active; if it stops renewing, another replica's attempt to acquire it succeeds and it becomes active instead.

Operating etcd: the fsync tax

The property that makes etcd's guarantees real — a write isn't acknowledged until it's durably on a quorum of disks — is also its sharpest operational edge. Every write requires an fsync to the leader's write-ahead log, then replication and an fsync on a quorum of followers, before the client gets an acknowledgment. That means etcd's write latency is bounded below by disk fsync latency — not network, not CPU. The commonly cited operating target is keeping etcd_disk_wal_fsync_duration_seconds p99 under 10ms; well above that, followers start missing heartbeats and triggering elections that have nothing to do with an actual failure.

The classic production footgun Network-attached storage (NAS, SAN, network block storage like some Ceph RBD configurations) adds an extra, less predictable network hop to what should be a local fsync — exactly the kind of latency spike Raft's heartbeat and election timers are sensitive to. The standard guidance is blunt: dedicated local SSD/NVMe for etcd's data directory, not shared with other heavy I/O, and sized generously (3, 5, or 7 members — 5 is the common production sweet spot for surviving 2 failures without paying too much extra write-latency tax for a larger quorum). Regular etcdctl snapshot save backups (commonly every 6–12 hours) round out the operational checklist, since Raft's replication protects against node loss, not against an operator deleting the wrong key.

Why it graduated, and where you see it

etcd graduated CNCF in November 2020 on the strength of exactly this track record: it had become the de facto canonical strongly-consistent store underneath cloud-native infrastructure generally, not just Kubernetes. CoreDNS has a native etcd plugin for serving DNS records out of an etcd keyspace independent of any cluster; Cloud Foundry's Diego scheduler used it as its coordination store; and the etcd-io/raft library underneath it all was factored out and is reused directly by CockroachDB, TiDB, Project Calico, and Flannel. When you see "Raft" named in an unrelated project's architecture doc, there's a decent chance it's this exact library.

Often confused with ZooKeeper and Consul solve the same class of problem (distributed coordination/consensus) and predate or parallel etcd — the meaningful difference for a Kubernetes engineer isn't "which is more correct," it's that etcd's gRPC API, MVCC revision model, and watch semantics are what Kubernetes' entire API machinery (resourceVersion, watches, leases) was built to match. Also worth separating: etcd is not a general-purpose database — no secondary indexes, no complex queries, deliberately small feature surface — because its one job is being the consistent source of truth a control plane can trust completely.
