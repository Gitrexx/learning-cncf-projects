CRI-O

The containerd deep-dive showed a general-purpose runtime daemon that Kubernetes happens to be one client of, alongside Docker, BuildKit, and nerdctl. CRI-O flips that premise: it exists for exactly one caller. There's no CLI to build images, no plugin ecosystem for non-Kubernetes clients, no API beyond the one the kubelet already speaks. If containerd is a general-purpose runtime that happens to serve Kubernetes well, CRI-O is what you get when you start from the CRI spec and add nothing else.

Where this fits CRI-O graduated CNCF in July 2023, joining containerd and Kubernetes itself at the top maturity tier. It's a CRI implementation only: the kubelet talks to it over /var/run/crio/crio.sock exactly the way it talks to containerd, and CRI-O turns those calls into running containers via an OCI-compliant runtime underneath. It's the default runtime for Red Hat OpenShift and a common choice anywhere an operator wants the smallest, most tightly-scoped runtime surface a Kubernetes node can run.

The problem: why build a second CRI runtime at all

CRI-O's origin predates containerd's CRI plugin. It started life in 2016 inside Red Hat as OCID, then moved under the kubernetes-incubator GitHub organization and was renamed CRI-O — "CRI" plus "O" for OCI, spelling out its entire scope in the name. At the time, Docker's monolithic dockerd was the only mainstream way to run containers, and Kubernetes talked to it through an out-of-tree adapter called dockershim that translated CRI calls into Docker Engine's own API — an extra hop, extra surface area, and extra failure modes for functionality (image building, Docker networking, the Docker CLI) that a kubelet never uses.

CRI-O's premise: if the kubelet only ever needs to run Pods and containers via a fixed, versioned gRPC interface, build a runtime that implements exactly that interface and stops. No image builder, no non-CRI API, no ambition to be a general-purpose developer tool. Every dependency it does carry — image handling, storage, container execution — is a shared library also used by Podman and Buildah, rather than code invented just for CRI-O. The result is a much smaller codebase than containerd's, a small idle memory footprint (tens of megabytes for the daemon itself), and a correspondingly smaller attack surface: fewer features means fewer things that can go wrong or be exploited.

Architecture: one daemon, one job, a monitor per container

Like containerd, CRI-O runs as a single daemon per node (crio) that the kubelet reaches over a Unix socket. But where containerd is built as a plugin host with a general-purpose client API and CRI as one plugin among several, CRI-O is the CRI service — there's no broader API surface underneath it that CRI happens to sit on top of.

CRI-O composes a small set of shared libraries rather than reinventing them:

  • containers/image resolves and pulls image manifests and layer blobs from a registry — the same library Podman and Buildah use to pull images.
  • containers/storage is CRI-O's equivalent of containerd's content store plus snapshotter combined: content-addressed layer storage with pluggable drivers, defaulting to overlayfs on Linux, that turns a stack of image layers into a mounted root filesystem plus one writable layer per container — the same layered, copy-on-write model containerd uses, implemented by a library shared across the Podman/Buildah/CRI-O family instead of built specifically for this one project.
  • CNI (Container Network Interface) plugins configure each Pod's network namespace — the same CNI plugin chain a cluster already has installed for its CNI-based networking, just invoked by CRI-O instead of by containerd's CNI integration.

conmon: a supervisor process per container, not a shim

containerd's answer to "who out-lives runc and stays the container's real parent process" is the shim (containerd-shim-runc-v2), talking back to containerd over ttrpc. CRI-O's answer is a lighter-weight, purpose-built C binary called conmon — "container monitor." CRI-O launches one conmon process per container; conmon then execs the OCI runtime (runc or crun) to actually create the container process per the OCI runtime spec, and once that runtime exits, conmon stays behind as the container's parent.

conmon's job is narrower than a containerd shim's: hold the container's stdio pipes and write its logs to disk (so crio restarting doesn't sever log capture), serve exec/attach connections, detect an OOM kill, and record the exit code to a file when the container dies — all without needing a full task-control RPC protocol, because CRI-O only needs conmon to answer to CRI-O, not to arbitrary other clients the way a containerd shim needs to answer over a stable, general-purpose Task API. Same "don't let the container die with the daemon" property as containerd's shim, deliberately built with a smaller surface.

runc vs. crun CRI-O's default OCI runtime shipped as runc (Go) for years, matching containerd, but recent CRI-O releases (1.35+) default to crun — a much smaller runtime written in C by the containers/ community, benchmarked at tens of milliseconds faster container start than runc with a lower memory footprint. Either one, or a VM-backed runtime like Kata Containers, can be configured per RuntimeClass: CRI-O reads runtime_handler off the CRI request and maps it to a configured runtime binary in crio.conf, the same "different Pods, different isolation, one node" trick containerd does via its own runtime-handler-to-shim table.

CRI request flow: same calls, different internals

Because both runtimes implement the identical CRI gRPC surface, the kubelet's call sequence looks the same either way: RunPodSandbox stands up the Pod's shared network namespace via a minimal sandbox, then CreateContainer and StartContainer land each container in it. Why it graduated, and where you see it

CRI-O graduated CNCF in July 2023. Its clearest production home is Red Hat OpenShift, which has shipped CRI-O as its only supported container runtime since OpenShift 4.0 — a large part of why CRI-O gets the hardening and real-world mileage a niche project wouldn't. Its release numbering still tracks Kubernetes minor versions (a CRI-O 1.31.x pairs with Kubernetes 1.31) and it follows the same n-2 version-skew policy Kubernetes itself uses for feature graduation, even though patch releases are cut independently and features can be backported off that cadence. Anywhere an operator wants the minimum runtime attack surface a node can run — security-conscious platforms, regulated environments, anyone who'd rather not carry code paths a kubelet never exercises — CRI-O is the default answer.

Often confused with containerd is the other production-grade CRI runtime, not a predecessor or successor to CRI-O — both are CNCF-graduated, both are common default choices on managed Kubernetes, and the difference is scope (general-purpose daemon vs. CRI-only) rather than maturity. Docker isn't a runtime choice for the kubelet at all anymore; dockershim was removed in Kubernetes 1.24, and Docker Engine itself just runs on containerd underneath. runc and crun aren't competitors to CRI-O — they're the OCI runtime executables that CRI-O's conmon (and containerd's shim) exec to do the actual namespace/cgroup work.
