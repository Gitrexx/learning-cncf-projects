containerd

The Kubernetes deep-dive drew a firm line at the kubelet: it schedules Pods, it enforces probes, it reports node status — but it never actually creates a Linux process, mounts a root filesystem, or pulls an image byte. Something below the kubelet has to do that literal work, on every node, for every container. That something is containerd: the daemon that turns "run this image" into a running process, and does it the same way whether the caller is kubectl-driven Kubernetes, plain Docker, or a CI pipeline building images.

Where this fits containerd graduated CNCF in February 2019. It's the industry-standard, daemon-based container runtime: it manages image transfer and storage, container execution and supervision, low-level storage, and network attachments on a single host. Docker has run on top of containerd since 2017 (Docker donated it to CNCF); Kubernetes' kubelet talks to it directly via containerd's built-in CRI plugin; and tools like nerdctl, BuildKit, and Firecracker-containerd all build on the same client API. If you've ever run crictl or seen /run/containerd/containerd.sock in a node's process list, you were looking at this project.

The problem: one runtime, many callers, no duplicated work

Before 2017, "Docker" meant one monolithic daemon (dockerd) that handled the CLI, image builds, networking, volumes, and low-level container execution all in one process — convenient for a single-user workstation, but awkward for anything else that wanted just the "run a container" part without dragging in the rest. Meanwhile the OCI (Open Container Initiative) had standardized what a container image and a container runtime actually are, so that standard could be implemented once and reused everywhere instead of every tool inventing its own container execution logic.

containerd is the result of pulling that "run a container" core out of dockerd into its own daemon, hardening it, and exposing it over a stable gRPC API so any client — Docker, Kubernetes, a CI system — can drive the same execution engine. It deliberately does not do the things a developer-facing tool needs: no image building, no CLI UX, no compose files. It focuses on being embedded into a larger system, which is exactly what both Docker and the kubelet do to it.

Architecture: a plugin daemon behind one gRPC API

containerd runs as a single long-lived daemon on each node, exposing a gRPC API over a Unix socket (/run/containerd/containerd.sock). Internally, almost everything — the content store, every snapshotter, the runtime, the CRI service itself — is a plugin, registered under an ID like io.containerd.grpc.v1.cri or io.containerd.snapshotter.v1.overlayfs and independently configured in /etc/containerd/config.toml. That plugin boundary is what let the CRI implementation move from an out-of-tree adapter into containerd's own process without touching the storage or runtime layers underneath it.

Multiple, mutually distrusting clients can share one containerd daemon because every gRPC call carries a namespace — not a Linux kernel namespace, but a containerd-specific tenancy label. Docker uses the moby namespace; Kubernetes uses k8s.io. Objects (images, containers, snapshots) in one namespace are invisible to another, so Docker and the kubelet can run against the very same containerd process on a node without ever seeing each other's containers.

Content store and images: everything is a content-addressed blob

An OCI image is not one file — it's a tree of blobs addressed purely by the digest of their content: a top-level manifest listing a config blob and an ordered list of filesystem layer blobs, each one a gzipped tarball diff. containerd's content store is nothing more than these blobs on disk, keyed by digest, plus a small metadata database (a boltdb file) recording what refers to what. Pulling an image is just: resolve a tag to a manifest digest, then fetch every blob the manifest references that isn't already present locally — which is exactly why two images sharing a base layer only ever store that layer once.

Snapshotters: layers become a real filesystem

A downloaded layer is an inert tarball; a running container needs an actual mounted root filesystem. That translation is the job of a snapshotter — containerd's pluggable abstraction over "stack these layers into a filesystem a process can chroot into." The default on Linux is overlayfs, mirroring Docker's overlay2 storage driver, though devmapper, btrfs, zfs, and others exist as plugins for hosts where overlayfs isn't available or desirable.

Unpacking works layer by layer, bottom to top: for each layer, the snapshotter opens a new active snapshot on top of the previous one, containerd's differ applies that layer's tarball diff into it, and the snapshotter commits it into a read-only snapshot that the next layer builds on. The result is a chain of read-only lowerdirs. Starting a container from that image adds one more active snapshot on top — a writable upperdir — and overlayfs presents the whole stack merged as a single mount. Every byte the container writes lands only in that top writable layer; the image layers underneath stay untouched and shareable across every other container started from the same image.

Four calls, four API layers: Pull fills the content store; unpacking drives the snapshotter's Prepare/Commit calls, one per layer; Create adds the container's own writable layer and a container object; Task.Start hands the merged rootfs to a shim, which execs runc.

Tasks, shims, and runc: what actually calls clone()

containerd itself never becomes the parent process of a container — that would mean every running container dies if containerd crashes or gets upgraded. Instead, a container object (image reference, OCI runtime spec, rootfs mount) is a separate thing from a task, the live, running instance of it. Starting a task spawns a small, per-container supervisor process called a shim — containerd-shim-runc-v2 by default — which is reparented to be its own process group leader and which in turn execs runc to do the actual namespace, cgroup, and pivot_root work of creating the container process, per the OCI runtime spec. runc exits once the container is running; the shim stays behind as the long-lived parent, reaping the container's exit status and exposing task control (pause, exec, kill, wait) over a lightweight ttrpc protocol back to containerd.

Why a shim per container, not one shim for everything Because the shim — not containerd — is the container's real parent process, restarting or upgrading the containerd daemon doesn't touch any already-running container: each shim just keeps supervising its one container and reconnects to the daemon when it comes back. This is the same "control plane restart shouldn't kill running work" property Kubernetes gives you at the Pod level, reproduced one layer down at the process level.

The shim v2 API is also how containerd supports runtimes other than runc without touching its own code: a runtime handler just names a different shim binary. Kata Containers' shim starts a lightweight VM instead of a plain process; gVisor's runsc intercepts syscalls in a user-space kernel instead of relying on the host kernel's namespaces directly. From containerd's point of view both are still "some shim implementing the Task API" — the isolation model underneath is the handler's problem, not containerd's.

The CRI plugin: how the kubelet actually talks to containerd

Since containerd 1.1, the CRI implementation has shipped inside containerd itself as a built-in plugin (io.containerd.grpc.v1.cri, reorganized as io.containerd.cri.v1.* plugins in containerd 2.x) rather than as a separate process — this is what let Kubernetes drop dockershim entirely (removed in Kubernetes 1.24) with containerd as a fully first-class, no-adapter-needed CRI runtime. The kubelet talks CRI gRPC straight to containerd.sock: RunPodSandbox to stand up a Pod's shared network namespace (backed by a minimal "pause"-style sandbox container), then CreateContainer/StartContainer for each container in that Pod, each one landing on the exact pull → unpack → create → start path above. The CRI plugin's own config lives under [plugins.'io.containerd.cri.v1.images'] and [plugins.'io.containerd.cri.v1.runtime'] in config.toml — including the runtimes table that maps a Kubernetes RuntimeClass name to a concrete shim, which is exactly how one node can run ordinary Pods on runc and sensitive Pods on Kata or gVisor side by side.

Config versions matter containerd's CRI plugin config format has evolved (version 2 became the default in 1.x; version 1 was removed, and version 3 became the only supported format in containerd 2.0). Upgrading a node's containerd major version without checking config.toml's version field is a common way to end up with a kubelet that can't create pods after an upgrade.

Garbage collection: leases keep content alive

Content blobs and snapshots are only useful as long as something references them, and containerd runs a mark-and-sweep garbage collector to reclaim anything that isn't. A lease is the mechanism that keeps a blob or snapshot alive across that sweep even while no container currently uses it — the same "explicit keep-alive object" shape as an etcd lease, just guarding disk content instead of keys. Delete an image and every layer only it referenced becomes collectible on the next GC pass; delete it while another image shares that layer, and the shared blob survives untouched.

Why it graduated, and where you see it

containerd graduated CNCF in February 2019, having already become the runtime underneath Docker itself (since the 2017 split) and the default, most widely deployed CRI runtime across managed Kubernetes offerings. Beyond running containers directly, its client library and content/snapshot model get reused as infrastructure by other tools: BuildKit builds images on top of it, nerdctl is a Docker-CLI-compatible frontend for it, and Firecracker-containerd swaps in a microVM-backed shim to run containers as Firecracker VMs.

Often confused with Docker is not a competitor to containerd anymore — Docker Engine runs on top of containerd today, adding the CLI, image build, networking, and compose UX that containerd deliberately leaves out. CRI-O (the next deep-dive) is a different runtime that also implements CRI, but was built to do nothing except what Kubernetes' CRI needs, whereas containerd is a general-purpose runtime that Kubernetes is one of several consumers of. And runc isn't a competitor to either — it's the OCI runtime executor that both containerd's shims and CRI-O actually shell out to underneath.
