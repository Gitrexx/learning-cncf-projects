Kubernetes

You already run Deployments, write Services, and debug CrashLoopBackOffs. This deep-dive isn't "Kubernetes 101" — it's the one piece of scaffolding the other 35 deep-dives in this landscape lean on. Every other CNCF graduated project either extends Kubernetes (Cilium, Istio), plugs into it through a defined interface (containerd via CRI, Rook via CSI), or runs on top of it (Argo, Dapr, Knative). To place any of them correctly, you need a precise mental model of where Kubernetes' job ends — not the Pods-and-Services surface, but the control loop underneath it.

Where this fits Kubernetes is CNCF's founding project — donated by Google in 2014, the first project accepted into CNCF's inaugural cohort in 2016, and the first project CNCF ever graduated, in March 2018. Every section of this landscape after this one assumes Kubernetes as the substrate.

The problem before Kubernetes

Kubernetes descends from Google's internal Borg system, which scheduled and managed containers across Google's fleets for over a decade before Kubernetes existed. The problem it solves predates containers: given a pool of machines and a set of workloads with resource needs, health requirements, and scaling targets, continuously keep the right number of the right workloads running on the right machines, without a human re-running scripts every time something crashes, a node dies, or demand changes.

Pre-Kubernetes container orchestration (early Docker Swarm, Mesos + Marathon, home-grown schedulers) mostly treated this as a one-shot placement problem: place container, done. What made Kubernetes different — and is the single idea to hold onto through this whole page — is that it treats orchestration as a closed reconciliation loop running forever, not a placement decision made once.

Two control planes: control plane and node

A Kubernetes cluster splits cleanly into two tiers. The control plane makes decisions and holds all state; it never runs your workloads. Nodes execute workloads and know almost nothing about the cluster as a whole — each node only cares about the Pods assigned to it. Control plane components

  • kube-apiserver — the only door into the cluster. Every read and write, from kubectl, from controllers, from kubelets, goes through it as a REST/gRPC front end over etcd. It's stateless and horizontally scalable; it does authentication, authorization (RBAC), and runs admission controllers before anything is persisted.
  • etcd — the strongly consistent key-value store (its own CNCF graduated project, using Raft consensus) that holds all cluster state. Nothing but the API server ever talks to it directly.
  • kube-scheduler — watches the API server for Pods that exist but have no node assigned, picks a node for each via filtering + scoring, and writes that decision back through the API server.
  • kube-controller-manager — runs dozens of independent control loops (Deployment, ReplicaSet, Node, Job, and more) bundled into one process, each watching one slice of desired vs. actual state and reconciling it.

Node components

  • kubelet — the agent on every node. It watches the API server for Pods assigned to its node, drives the container runtime through the CRI to actually run them, runs liveness/readiness probes, and reports status back up.
  • kube-proxy — maintains the network rules (iptables, IPVS, or nftables, depending on mode) on each node that implement Service virtual IPs, watching Endpoints / EndpointSlices for changes.
  • container runtime — containerd or CRI-O, reached via CRI; it's what actually pulls images and starts containers. (Its own deep-dive is next in this section — Kubernetes deliberately doesn't implement this itself.)

The idea underneath everything: declarative reconciliation

Every Kubernetes object you write — a Deployment, a Service, a PersistentVolumeClaim, a CRD instance — has a spec (what you want) and a status (what's currently true). A controller is a loop that does exactly one thing, forever: 

for {
    observed := getCurrentState(object)   // e.g. list actual running Pods
    desired  := object.Spec               // e.g. replicas: 3
    diff     := desired - observed        // e.g. 1 Pod short
    if diff != 0 {
        act(diff)                         // e.g. create one more Pod
    }
    wait for next change or resync period
}

Two properties of this loop matter more than they first look like they should:

Level-triggered, not edge-triggered A controller doesn't reason about the event that woke it up ("a Pod died"). It always re-reads the full current state and compares it to the full desired state. This is why Kubernetes self-heals from almost anything, including missed or duplicate events, a controller crashing and restarting, or the API server being unreachable for a while — the next reconcile just re-derives the right action from scratch. Contrast with edge-triggered systems, where a missed event means the action never happens.

Controllers don't poll the API server in a busy loop, either. They use etcd's watch mechanism through the API server: register interest in a resource type, get pushed only the deltas since your last known resourceVersion, and fall back to a periodic full list ("resync") as a safety net in case a watch silently dropped events. This is the same watch/resourceVersion pattern you'll see reused by Argo CD, Flux, and virtually every operator in the ecosystem — Kubernetes didn't just standardize the object model, it standardized how you watch it.

The object model is controllers, not magic

Once you see the reconciliation loop, the familiar object hierarchy stops looking like special Kubernetes behavior and starts looking like ordinary controllers watching ordinary objects:

ObjectWho reconciles itWhat "reconcile" means here
Deploymentdeployment-controllerEnsure the right ReplicaSet exists for the current template, roll old ReplicaSets down and the new one up.
ReplicaSetreplicaset-controllerEnsure exactly spec.replicas Pods matching the label selector exist; create or delete Pods to close the gap.
Podkubelet (per-node) + schedulerGet it a node (scheduler), then keep its containers running to match its spec (kubelet).
Serviceendpoint-controller + kube-proxyTrack which Pods match the selector as Endpoints; kube-proxy turns that into local routing rules.

This is also exactly the pattern every CRD + operator in the ecosystem follows (Rook for Ceph, cert-manager for certificates, Crossplane for cloud infrastructure): define a custom spec/status object, write a controller that reconciles it. Kubernetes' real export isn't Pods — it's this control-loop pattern, generalized via CRDs so anything can be modeled and reconciled the same way.

Where Kubernetes' job ends

Because this landscape is organized around "who does what job around the anchor," it matters to be precise about what's actually implemented inside Kubernetes versus what Kubernetes only defines an interface for for other projects to fill in:

LayerKubernetes definesWho actually does it
Container executionCRI (gRPC interface)containerd, CRI-O
Pod networkingCNI (interface)Cilium, and other CNI plugins
Storage provisioningCSI (interface)Rook, cloud-provider CSI drivers
Service-to-service traffic mgmtnothing — sidecar injection is opt-inIstio, Linkerd
Cluster DNSa required cluster add-onCoreDNS
Metrics & alertinga metrics API surfacePrometheus
Continuous delivery from Gitnothing — it's a client of the APIArgo CD, Flux
Package templatingnothing — renders plain manifestsHelm

This is the recurring shape of the whole CNCF landscape: Kubernetes defines a narrow interface (CRI, CNI, CSI) or leaves a layer alone entirely, and a graduated project fills it in. Keep this table in mind — you'll see almost every row become its own deep-dive.

Why it graduated, and where you actually see it

Kubernetes graduated CNCF's maturity process first (March 2018) essentially by definition — the process itself was modeled around a project of Kubernetes' scale and governance already having proven multi-vendor adoption, a diverse committer base, and production use at organizations beyond its creator. Today it's the substrate under every major managed offering — EKS, GKE, AKS — and under most private-cloud and on-prem platform teams' internal platforms. When you see "cloud native" used as a near-synonym for "runs on Kubernetes," this is why: it became the portable abstraction that made workloads move between clouds and data centers without a rewrite, which is the whole reason CNCF exists.

Often confused with "Kubernetes" the project vs. "a Kubernetes distribution" (OpenShift, Rancher, k3s, EKS/GKE/AKS) — the distributions bundle Kubernetes with their own defaults, add-ons, and support model, but all run the same upstream control-plane code and API. Nothing in this table above is a Kubernetes "feature" you're missing — it's a deliberate interface boundary.
