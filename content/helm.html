Helm

Every deep-dive so far has been about running one container correctly. Helm is about a different problem: a real application is never one Deployment. It's a Deployment, a Service, a ConfigMap, maybe an Ingress and a couple of Secrets and an HPA — a dozen-plus YAML files that all have to agree with each other (the same labels, the same name prefix, the same image tag) and that need to move together as one versioned, installable, upgradeable, rollback-able unit. Helm is the tool that turns that pile of YAML into a single package you can install, upgrade, and undo with one command.

Where this fits Helm graduated CNCF in April 2020 and is, by a wide margin, the most widely used way to package and ship applications onto Kubernetes — it's often called "the package manager for Kubernetes." Helm doesn't replace kubectl apply — under the hood it still ends up calling the Kubernetes API the same way — it replaces hand-managing the *set* of manifests an application needs, their variable substitution, and their lifecycle as a single named unit.

The problem: raw manifests don't compose or version

Plain kubectl apply -f works fine for one file. It stops working well once an application is more than a couple of resources deployed to more than one environment. Three things get painful fast:

  • Parameterization. Staging and production need different replica counts, image tags, and resource limits from the same manifests. Without a templating layer, that means maintaining near-duplicate YAML per environment, or a pile of sed/CI string substitution around it.
  • Grouping. Kubernetes has no native concept of "these twelve objects are one application." kubectl applies and deletes objects individually; nothing tracks that a Deployment, a Service, and a ConfigMap were installed together and should be upgraded, rolled back, or removed together.
  • History. If an upgrade goes bad, kubectl apply has no built-in "what was running two versions ago, and can I get back to exactly that" — you'd need to have kept your own copies of every previous manifest set.

Helm's answer is a chart — a directory (or packaged .tgz) containing templated manifests and default configuration — and a release — what you get when you install a chart with a specific set of values into a specific cluster. One helm install creates every resource in the chart as one release; one helm upgrade changes them together; one helm rollback reverts all of them to a prior revision in one shot.

Architecture: a client talking straight to the Kubernetes API

This is the single most important thing to know about Helm's architecture, because it's the one thing that changed between major versions in a way people still bring up: Helm 2 ran an in-cluster server component called Tiller that held broad RBAC permissions and did the actual create/update/delete work on the client's behalf. Tiller was a real security problem — a single component with cluster-wide write access that every Helm user talked to — and it made per-user, per-namespace permission scoping awkward.

Helm 3 removed Tiller entirely. Helm today is just a client — a CLI binary (or the equivalent Go SDK) that renders templates locally and then talks directly to the Kubernetes API server, authenticated as whatever identity your kubeconfig already represents. There's nothing else to install in the cluster and nothing extra to secure: Helm's access is exactly your kubectl access, enforced by the same RBAC rules Kubernetes already applies to everyone else.

So what tracks releases, if there's no server? State has to live somewhere so helm upgrade and helm rollback know what's currently installed and what came before. Helm 3 stores each release revision as a Kubernetes Secret in the release's namespace (Helm 2 used ConfigMaps; Secrets became the default once Kubernetes added encryption-at-rest for them), named sh.helm.release.v1.<release-name>.v<revision>. Each Secret holds the fully rendered manifest, the values used, and metadata, gzip-compressed and base64-encoded to stay well under etcd's per-object size limit. helm history just lists these Secrets; helm rollback reads an older one and re-applies it.

Anatomy of a chart

A chart is a directory with a fixed, conventional layout. Helm doesn't require every file, but this is the shape almost every real chart takes: 

mychart/
  Chart.yaml          # name, version, appVersion, dependencies metadata
  values.yaml          # default configuration, overridable at install time
  charts/               # vendored subcharts (dependencies), each a chart itself
  templates/
    deployment.yaml     # Go-template manifests, rendered against values
    service.yaml
    ingress.yaml
    _helpers.tpl         # named template snippets (helpers), no manifest of its own
    NOTES.txt             # post-install usage text, also templated
  .helmignore            # patterns excluded when packaging

Chart.yaml carries the chart's own identity — version is the chart's version (SemVer, bumped every time templates or defaults change), separate from appVersion, which just documents the version of the software the chart deploys. A chart for nginx might be at chart version: 4.2.0 while still deploying appVersion: 1.25 — chart iteration and app releases move on independent clocks.

Templating: Go templates plus Sprig, resolved against layered values

Every file under templates/ is rendered as a Go template, extended with the Sprig function library for things Go templates don't do natively — string manipulation, default values, type conversion, even cryptographic helpers for generating TLS certs inline. A template references configuration through a fixed set of built-in objects:

  • .Values — the merged configuration (see precedence below).
  • .Release — .Release.Name, .Release.Namespace, .Release.Revision — facts about this specific install.
  • .Chart — the contents of Chart.yaml (.Chart.Name, .Chart.Version...).
  • .Files, .Capabilities — access to non-template files in the chart, and to what the target cluster supports (API versions, Kubernetes version).

.Values is a merge, not a single source, resolved in increasing precedence: the chart's own values.yaml defaults, then a parent chart's values for any subchart, then -f/--values files passed on the command line (later files win over earlier ones), then individual --set key=value overrides, which win over everything. This is exactly why a chart works out of the box with zero configuration (sane defaults in values.yaml) but is fully overridable per environment without ever editing the chart itself.

The release lifecycle

Helm's core verbs all operate on a release as a whole, not on individual objects:

  • helm install <release> <chart> — render templates against values, create every resulting object, save revision 1.
  • helm upgrade <release> <chart> — re-render with new values and/or a new chart version, diff against what's live, apply the changes, save the next revision. Deleted resources (removed between chart versions) are cleaned up too.
  • helm rollback <release> <revision> — re-apply a previously stored revision's rendered manifests as a new revision (rollback is itself a new upgrade, not a time-travel delete).
  • helm uninstall <release> — delete every object the release owns and, by default, its revision history.

Because every revision is a snapshot, helm history <release> shows the full chain, and helm diff (a popular plugin, not built in) can show exactly what an upgrade would change before you run it.

Dependencies and the chart ecosystem

Charts can depend on other charts — a web app chart might declare a dependency on a Redis or PostgreSQL chart rather than reinventing that YAML. Dependencies are declared in Chart.yaml: 

dependencies:
  - name: redis
    version: "18.x.x"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled

helm dependency update resolves and vendors matching chart versions into charts/. condition/tags let a parent chart toggle a subchart on or off through its own values.yaml (exactly what redis.enabled: false in a parent's values would do), and alias lets the same dependency be included more than once under different names.

Charts are distributed two ways. The original mechanism is a chart repository: a plain HTTP server hosting packaged .tgz charts plus an index.yaml listing them, added locally with helm repo add. Since Helm 3.8, charts can also be pushed to and pulled from any OCI-compliant registry — the same registries that store container images — as a first-class artifact type, addressed like oci://registry.example.com/charts/mychart. OCI distribution is increasingly the default: it reuses registry infrastructure teams already run and secure (auth, replication, signing) instead of a Helm-specific HTTP server, and several major chart publishers have moved or are moving their catalogs to OCI registries as classic index-based repos see less investment.

Why it graduated, and where you see it

Helm graduated CNCF in April 2020, the same class as CoreDNS. Its ubiquity comes from being the default answer to "how do I install this open-source thing onto my cluster" — helm install prometheus prometheus-community/kube-prometheus-stack or helm install argocd argo/argo-cd is how most people first bring up major CNCF projects, including several others on this list. Vendors and open-source projects ship a Helm chart as their primary Kubernetes install path essentially by convention now, and Helm charts are frequently the packaging format platform teams standardize internal application templates on top of.

Often confused with Kustomize solves an overlapping but distinct problem: it takes plain, valid YAML and layers patches on top per environment — no templating language, no values file, nothing to render, which fits GitOps workflows that want to diff plain manifests. Helm templates generate YAML from a non-YAML templating language and a values file; Kustomize only ever patches YAML that was already valid YAML. Many teams use both — Helm to package an upstream chart, Kustomize (or Helm's own post-renderer hook) to patch its output. Operators (built with the Operator SDK / Kubebuilder, often distributed via OLM) are a different mechanism again: an Operator is running code, a controller reconciling a CRD continuously, versus Helm's one-shot template-and-apply with no ongoing controller of its own. A chart can install an Operator; the Operator then does its own ongoing reconciliation independent of Helm.
